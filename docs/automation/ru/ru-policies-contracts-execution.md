## Назначение

Определить единый стандарт документирования:

- политик,
- операционных контрактов,
- и фактического исполнения.

## Область

Применяется ко всем governance-артефактам в `docs/automation/`:

- политикам,
- контрактам,
- execution evidence.

## Предусловия

- Назначен владелец документа.
- Определен риск (`low|medium|high`).
- Известна связка с методом/API.
- Планируется обновление `docs/knowledge-registry.json` в том же PR.

## Процедура

Каждый автоматизированный процесс должен иметь:

1. Политику (что разрешено/запрещено).
2. Контракт (кто и за что отвечает).
3. Доказательства исполнения (что реально произошло).

Без этих трех артефактов процесс не считается automation-ready.

### Стандарт для политик

Документ политики должен содержать:

- область действия,
- модель рисков (`low`, `medium`, `high`),
- матрицу согласований,
- запрещенные действия,
- триггеры эскалации,
- процесс исключений.

### Стандарт для контрактов

Контракт должен содержать:

- участников (человек/агент),
- зоны ответственности,
- точки передачи,
- SLA/SLO,
- критерии приемки,
- владельца rollback.

### Стандарт для доказательств исполнения

Каждый execution-лог должен содержать:

- execution ID,
- источник запроса,
- решение политики (`allowed`/`denied` + причина),
- выполненные действия,
- артефакты (логи/ссылки/файлы),
- итоговый статус и sign-off.

## Сценарии отказа

- Нет одного из трех обязательных артефактов.
- В реестре отсутствуют owner/status/review даты.
- Контракт не содержит критериев приемки или владельца rollback.
- Execution evidence без решения policy (`allowed|denied`) и причины.

## Эскалация

- При неполном наборе артефактов статус остается `draft`, `agent_ready=true` запрещен.
- Для high-risk пробелов требуется ревью Safety/Audit.
- Создается remediation-задача с owner и due date.
